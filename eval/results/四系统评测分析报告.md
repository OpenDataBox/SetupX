## 2. 评测方法论

### 2.1 Prosecutor + Judge 诉讼模型

每个系统的评测遵循统一流程：

1. **构建阶段**：用系统产出的 Dockerfile（+ install.sh）构建 Docker 镜像
2. **检察官调查**：独立的 Prosecutor Agent 进入容器，检查核心依赖是否可导入、测试框架是否可用
3. **法官裁决**：Judge Agent 逐条审查检察官指控，进入容器验证依赖声明文件，区分"核心依赖"与"可选依赖"

核心设计理念：将"环境配置工具的自我报告"与"独立第三方核查"分离，避免自说自话。

### 2.2 裁决标准

- **not_guilty**：检察官未起诉，或法官驳回全部指控
- **guilty**：法官认定至少一条指控成立（通常是核心依赖确实不可导入）
- 法官会排除以下误报场景：检察官将 extras_require / optional-dependencies 误判为核心依赖；检察官引用了不存在的依赖声明位置

## 4. ExecAgent 评测框架 Bug 发现与修复

### 4.1 背景

ExecAgent 初版评测通过率仅 31.8%（14/44），显著偏低。经深入排查，发现评测脚本 `eval_execagent_full.py` 存在两个框架级 bug，导致约 5 个 repo 被错误判定为 guilty。

### 4.2 Bug 1：find_repo_dir() 回退路径错误

**问题**：当容器内 `.git` 目录不在预设搜索路径中时，`find_repo_dir()` 回退返回 `/workspace`。但 ExecAgent 的容器中通常没有 `/workspace` 目录，导致所有后续 `exec_run()` 命令以 `bash: line 1: cd: /workspace: No such file or directory` 失败。

**影响**：约 8 个 repo 受影响，Prosecutor 的所有 `python3 -c "import xxx"` 命令全部报错，但报错原因是"找不到工作目录"而非"依赖缺失"。

**修复**：
1. 扩展搜索路径（增加 `/opt/`, `/home/`, `/root/` 等）
2. 增加 `find / -maxdepth 4 -name '.git'` 全局搜索
3. 回退从 `/workspace` 改为 `/`（至少不会导致 cd 失败）

**修复效果**：python-control、smart_open、tmuxp 3 个 repo 翻转为 not_guilty。其余 5 个容器确实没有正确 clone 源码，guilty 判决正确。

### 4.3 Bug 2：_inject_install_env() 环境变量注入不完整

**问题**：原实现仅读取 `/tmp/_install_env.txt`（trap EXIT 快照），但很多 ExecAgent 的 install.sh 通过 `python3 -m venv`、`poetry install`、`uv sync` 创建了虚拟环境。快照中虽有 PATH 信息，但解析逻辑未能识别出 venv 路径。

**影响**：约 11 个使用 venv/poetry/uv 的 repo 受影响，Prosecutor 使用系统 python3 检查 import，看不到 venv 中已安装的包。

**修复**：替换为 `_inject_env_from_container()` 动态探测函数，按优先级激活容器内环境：
1. pyenv（最低优先级）
2. conda（中等优先级）
3. uv venv / poetry venv（高优先级）
4. 通用 .venv/venv 目录（最高优先级）

**修复效果**：molecule、poetry 2 个 repo 翻转为 not_guilty。其余 9 个的 venv 确实创建失败（如 sentry-python 缺少 python3-venv 包，`python3 -m venv .venv` 报错），guilty 判决正确。

### 4.4 修复后残留 guilty 的验证

手动抽查了多个修复后仍 guilty 的 repo，确认均为真实失败：

| repo | 验证结果 |
|------|----------|
| sentry-python | Dockerfile 未安装 python3-venv，`python3 -m venv .venv` 失败 |
| twine | 同上，venv 创建失败 |
| piccolo | 同上，venv 创建失败 |
| django-autocomplete-light | 容器内 pip list 仅有基础包，依赖确实未安装 |
| autograd | 源码路径错误，numpy 等依赖未安装 |

## 5. ExecAgent 低通过率根因深度分析

修复评测框架 bug 后，ExecAgent 仍有 18 个 guilty + 7 个 BUILD_ERR，通过率仅 43.2%。以下基于 18 个 guilty 案例的 Dockerfile、install.sh 和检察官指控进行逐案归因。

### 5.1 失败分类

#### 类型 A：源码根本不在容器中（9 个，占 guilty 的 50%）

| repo | Dockerfile 问题 |
|------|-----------------|
| lark | 只有 `WORKDIR /app/lark`，**没有 git clone** |
| fastapi-pagination | 同上，目录为空 |
| nonebot2 | 同上，目录为空 |
| pyftpdlib | 同上，目录为空 |
| platformio-core | 同上，目录为空 |
| briefcase | git clone 被 `\|\| exit 0` 吞掉失败 |
| memegen | 同上 |
| neuralforecast | 同上 |
| twine | 有 git clone 但目录仍为空（clone 失败被静默） |

以 lark 为典型案例，其完整 Dockerfile：

```dockerfile
FROM python:3.11-slim
WORKDIR /app/lark
CMD ["/bin/bash"]
```

没有 `git clone`、没有 `COPY`、没有任何获取源码的步骤。install.sh 里写了 `pip install -r test-requirements.txt`，但容器内根本没有这个文件。

#### 类型 B：依赖安装失败（7 个）

源码存在但 install.sh 执行后依赖仍缺失：

| repo | 具体原因 |
|------|----------|
| sentry-python | Dockerfile 用 debian:11 但未安装 python3-venv，install.sh 的 `python3 -m venv .venv` 失败 |
| piccolo | 同上，venv 创建失败 |
| datasets | install.sh 中 pip install 失败（依赖冲突或超时） |
| litgpt | 同上，7 项核心依赖（torch, lightning 等）全缺 |
| luigi | 同上，5 项核心依赖（tornado, tenacity 等）全缺 |
| river | 同上，scipy/pandas/numpy 全缺 |
| django-autocomplete-light | install.sh 执行后 pip list 仅 pip/setuptools |

#### 类型 C：基础环境损坏（2 个）

| repo | 具体原因 |
|------|----------|
| guardrails | Dockerfile 只做了 `apt-get update`，**没有安装 python3** |
| python-holidays | 最小化镜像，bash 不可用（exit_code=127） |

### 5.2 核心病因：`|| exit 0` 模式

几乎所有 ExecAgent 的 Dockerfile 都使用了这个模式：

```dockerfile
RUN git clone https://github.com/xxx/yyy . || exit 0
RUN apt-get install -y python3 python3-pip || true || { echo "Command failed"; exit 0; }
```

`|| exit 0` 意味着**任何命令失败都返回成功码**，Docker build 永远不会因为实际错误而中断。这导致：

1. **git clone 网络超时** → 静默跳过 → 容器内无源码 → 但镜像构建"成功"
2. **apt-get install 缺包** → 静默跳过 → python3-venv 缺失 → 但镜像构建"成功"
3. **ExecAgent 自评 SUCCESS** → 因为 Docker build exit_code=0

这是 ExecAgent **自评 88% SUCCESS 但第三方验证仅 43.2%** 的根本原因：`|| exit 0` 让自评形同虚设。

### 5.3 Dockerfile 与 install.sh 脱节

ExecAgent 的工作模式是：Agent 交互式地在容器中调试，最终产出 Dockerfile + SETUP_AND_INSTALL.sh 作为可复现产物。但产物质量存在严重问题：

1. **Dockerfile 缺少关键步骤**：Agent 在交互式会话中执行了 git clone，但写出的 Dockerfile 中遗漏了这一步（如 lark、fastapi-pagination 等 9 个 repo）
2. **install.sh 前提条件不满足**：install.sh 假设容器内已有源码和特定工具（如 python3-venv），但 Dockerfile 未安装这些前提
3. **错误被掩盖**：`|| exit 0` 让 Dockerfile build 永远成功，Agent 无法从 build 结果中发现自己遗漏了步骤

**对比 Ours 系统**：Setup Agent 在同一个容器中实时执行命令并验证结果（pytest 两步验证），每一步的成功/失败都有反馈。不存在"交互式调试"与"最终产物"之间的脱节。

### 5.4 小结

ExecAgent 43.2% 通过率的构成：

| 状态 | 数量 | 占比 |
|------|------|------|
| 真实通过（not_guilty） | 19 | 43.2% |
| 源码缺失（git clone 失败/遗漏） | 9 | 20.5% |
| 依赖安装失败 | 7 | 15.9% |
| 基础环境损坏 | 2 | 4.5% |
| Docker 构建失败 | 7 | 15.9% |
| **总计** | **44** | **100%** |

## 6. Ours 系统 pip 案例分析（系统已知局限）

### 6.1 背景

pip (pypa/pip) 在 Ours 系统中被 Prosecutor + Judge 判定为 guilty。**报告采用系统原始裁决（guilty），不做人工覆盖。** 以下分析仅记录调查发现，作为系统局限性的参考。

### 6.2 调查发现

人工调查发现此案例中 Prosecutor LLM 存在幻觉：

1. **cryptography 是可选测试依赖**：pip 的 `conftest.py` 明确设计为"缺少 cryptography 时自动 deselect 相关测试"，1702 个 pytest 测试在无 cryptography 环境下正常通过
2. **Prosecutor 的推理错误**：Step 28 执行 `pytest tests/lib/certs.py -v`，exit_code=0（成功），但 LLM 推理中声称"确认了 cryptography 缺失导致 ImportError"
3. **Prosecutor 忽视了自己发现的免责证据**：Step 16 已发现 conftest.py 中的 deselect 注释，但仍执行"核心依赖缺失必须起诉"的规则
4. **Judge 未独立复核**：盲信了 Prosecutor 的推断

### 6.3 启示

此案例暴露了 Prosecutor + Judge 模型的一个边缘风险：当 LLM 产生关于命令输出的幻觉时，Judge 如果不独立重跑命令就无法纠错。后续改进方向：强化 Judge 对 Prosecutor 引用证据的独立验证。

**但此案例不影响报告数据**——系统裁决即最终裁决，人工调查仅供参考。

## 7. EnvBench 环境变量注入缺陷：发现、根因与修复

### 7.1 问题发现

EnvBench 33 个新 repo 的首轮评测结果为 14 PASS / 19 FAIL（42% 通过率），guilty 案例数量异常偏高。

**异常信号**：大量 guilty 案例中，检察官的 `python3 -c 'import xxx'` 全部失败，但 SETUP_AND_INSTALL.sh 的内容显示依赖应该已经安装（在 conda 环境或 poetry venv 中）。

### 7.2 根因分析

通过进入多个 guilty 容器手动调查，发现根本原因是 **Docker exec 环境变量隔离**：

#### 7.2.1 环境丢失机制

EnvBench 的 Dockerfile 采用以下模式：

```dockerfile
RUN bash SETUP_AND_INSTALL.sh
```

SETUP_AND_INSTALL.sh 内部执行了：
```bash
pyenv install 3.11.7 && pyenv global 3.11.7
conda create -n autograd-env python=3.11
conda activate autograd-env && pip install -e .
poetry install
```

**关键问题**：`RUN` 层执行完毕后，`conda activate`、`pyenv global`、`poetry env` 设置的环境变量（PATH、CONDA_PREFIX、VIRTUAL_ENV 等）**不会持久化到容器运行时**。Docker 的每个 `RUN` 层是独立的 shell 进程，环境变量仅在该层有效。

当评测框架通过 `docker exec` 进入容器执行 `python3 -c 'import xxx'` 时，使用的是系统默认的 `/usr/bin/python3`（无任何 venv/conda 包），而不是 SETUP_AND_INSTALL.sh 中配置的 python 解释器。

#### 7.2.2 PATH 优先级叠加问题

部分 repo 同时使用 pyenv + conda（如 autograd），即使我们探测到了两个环境，激活顺序也至关重要：

```
错误顺序（conda 先、pyenv 后）：
PATH = /root/.pyenv/versions/3.11.7/bin:/opt/conda/envs/autograd-env/bin:...
→ python3 指向 pyenv 的裸 python（无 numpy）

正确顺序（pyenv 先、conda 后）：
PATH = /opt/conda/envs/autograd-env/bin:/root/.pyenv/versions/3.11.7/bin:...
→ python3 指向 conda env 的 python（有 numpy）
```

### 7.3 修复方案

在 `scripts/eval_envbench_33.py` 中加入 `_inject_env_from_container()` 函数，在 Prosecutor 调查之前执行容器内环境探测，按优先级从低到高依次激活 pyenv → conda → poetry → venv。

### 7.4 修复效果

| 阶段 | PASS | FAIL | 通过率 |
|------|------|------|--------|
| 修复前（33 repo） | 14 | 19 | 42% |
| 修复后（33 repo） | 26 | 6 (+1 PEND) | 79% |
| 合并旧 11 repo（44 repo） | 37 | 6 (+1 PEND) | 84% |

修复消除了 13 个假阳性（false guilty），其中：
- 12 个是纯环境变量丢失导致（Type A）：`python3` 指向系统 python，看不到 venv 里的包
- 1 个是 PATH 优先级错误导致（Type B）：pyenv shim 覆盖了 conda env 的 python

### 7.5 EnvBench 旧 11 repo 置信度问题

EnvBench 旧版 11 个 repo 的评测使用 `eval_envbench.py`（非 `eval_envbench_33.py`），其中 `_inject_bootstrap_env()` 逻辑较为简单，仅读取 `/tmp/_bootstrap_env.txt`。11/11 全部判定为 not_guilty，但置信度仅 40-50%——如果环境变量注入静默失败，Prosecutor 可能遗漏了本该发现的问题（假阴性）。

不过方向有利：如果存在误判，也是"放过了本该 guilty 的"（假阴性），不会虚增通过率。

## 10. 结论

### 10.1 系统排名与推荐

1. **Ours (+XPU) — 95.5%**：最高通过率。2 个失败中，plugin.video.netflix 是 Agent 提前退出的真实失败，pip 是 Prosecutor LLM 幻觉导致的争议案例（但报告采用系统原始裁决）。架构上天然免疫环境变量丢失问题。

2. **EnvBench — 84.1%**：修复环境变量注入后表现良好。残留 6 个 FAIL 中 5 个是真实失败，1 个（tmuxp）可能是环境探测残留问题。旧 11 repo 的评测置信度偏低，可能存在假阴性。

3. **R2R — 75.0%**：使用系统 python 避免了 venv 问题，评测结果可信度高。11 个失败 repo（6 FAIL + 5 BUILD_ERR）反映其 Dockerfile 生成质量不如 Ours。

4. **ExecAgent — 43.2%**：修复框架 bug 后从 31.8% 提升至 43.2%，但仍远低于其他三系统。18 个 guilty 经手动验证均为真实失败（install.sh 未执行、python3-venv 缺失、源码丢失等），**ExecAgent 自评 SUCCESS 率 88%（44/50）与第三方验证通过率 43.2% 之间的巨大差距**印证了独立裁决的必要性。

### 10.2 评测框架可靠性

本次评测共发现并修复了三类框架 bug：

| Bug 类型 | 影响系统 | 影响数量 | 修复效果 |
|----------|----------|----------|----------|
| Docker exec 环境变量丢失 | EnvBench | 13 个假阳性 | 通过率 42% → 84% |
| find_repo_dir 回退路径错误 | ExecAgent eval | 3 个假阳性 | — |
| _inject_install_env 不完整 | ExecAgent eval | 2 个假阳性 | 通过率 31.8% → 43.2% |
| Prosecutor LLM 幻觉 | Ours | 1 个争议案例（pip） | 不修正，保留系统原始裁决 |

**关键教训**：评测外部系统产出的 Docker 环境时，`docker exec` 新开的 shell 不继承 RUN 层或先前 exec 的环境变量，必须主动探测并注入。Ours 系统因架构设计（Agent 和 Prosecutor 共享同一进程上下文）天然免疫此类问题。

### 10.3 独立第三方裁决的价值

ExecAgent 的案例最能说明独立裁决的价值：其自评 SUCCESS 率为 88%（44/50），但经 Prosecutor + Judge 独立验证后降至 43.2%（19/44）。差距达 **45 个百分点**。如果仅依赖 Agent 自我报告，将严重高估环境配置的成功率。

同时，本次审计也证明了 Prosecutor + Judge 模型的一个已知局限：Prosecutor LLM 可能产生关于命令输出的幻觉，而 Judge 如果不独立重跑命令就无法纠错（pip 案例）。后续改进方向：强化 Judge 对 Prosecutor 引用证据的独立验证。
